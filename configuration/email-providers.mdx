---

Zero connects to external email providers through OAuth, which means your credentials are never stored by Zero — only the OAuth access and refresh tokens. Gmail is the primary supported provider and requires a Google Cloud project with the Gmail API and People API enabled. Microsoft Outlook support is implemented in the driver layer and will be available in a future release.

---

## Gmail (Google OAuth)

Connecting Gmail requires a Google Cloud project with OAuth 2.0 credentials. Follow these steps once per deployment — you can connect as many Gmail accounts as you like after setup is complete.

### Set up Google Cloud

<Steps>
  <Step title="Create a Google Cloud project">
    Open the [Google Cloud Console](https://console.cloud.google.com) and create a new project. Give it a recognisable name such as "Zero Email".

    <Tip>
      If you already have a project you want to use, you can skip this step and use the existing one.
    </Tip>
  </Step>

  <Step title="Enable the Gmail API">
    Go to [**APIs & Services → Enable APIs and Services**](https://console.cloud.google.com/apis/library) and enable the following two APIs:

    - [Gmail API](https://console.cloud.google.com/apis/library/gmail.googleapis.com)
    - [People API](https://console.cloud.google.com/apis/library/people.googleapis.com)

    Click **Enable** on each API page. Both are required — the People API provides profile information (name, photo) and the Gmail API provides inbox access.
  </Step>

  <Step title="Configure the OAuth consent screen">
    Go to [**APIs & Services → OAuth consent screen**](https://console.cloud.google.com/auth/overview).

    1. Select **External** as the user type (unless your project is in a Google Workspace organization).
    2. Fill in the required fields: **App name**, **User support email**, and **Developer contact email**.
    3. Click **Save and Continue** through the **Scopes** and **Test users** steps — you will add scopes via the credentials in the next step.

    <Note>
      While your app is in **Testing** mode, only email addresses you add as test users can sign in. You can stay in Testing mode for local development; you only need to publish the app if you plan to allow any Google account to sign in.
    </Note>
  </Step>

  <Step title="Add test users">
    In [**APIs & Services → Audience**](https://console.cloud.google.com/auth/audience), scroll to **Test users** and click **Add Users**. Enter the Gmail address you want to use for sign-in and click **Save**.

    Each team member who needs access during development must be added here.
  </Step>

  <Step title="Create OAuth 2.0 credentials">
    Go to [**APIs & Services → Credentials**](https://console.cloud.google.com/apis/credentials) and click **Create Credentials → OAuth client ID**.

    - **Application type:** Web application
    - **Name:** Zero (or any name)
    - **Authorized redirect URIs:** Add the URIs for your environment (see below)

    <Tabs>
      <Tab title="Local development">
        ```
        http://localhost:8787/api/auth/callback/google
        ```
      </Tab>
      <Tab title="Production">
        ```
        https://your-backend-domain.com/api/auth/callback/google
        ```

        Replace `your-backend-domain.com` with the actual domain where your backend (`VITE_PUBLIC_BACKEND_URL`) is hosted.
      </Tab>
    </Tabs>

    <Warning>
      The redirect URI must match **exactly** — including the protocol (`http` vs `https`), domain, port, and path. A mismatch results in a `redirect_uri_mismatch` error and blocks all sign-in attempts.
    </Warning>

    Click **Create**. A dialog shows your **Client ID** and **Client Secret**.
  </Step>

  <Step title="Add credentials to your .env file">
    Copy the Client ID and Client Secret into your `.env` file:

    ```env
    GOOGLE_CLIENT_ID=123456789-abcdefg.apps.googleusercontent.com
    GOOGLE_CLIENT_SECRET=GOCSPX-your_client_secret
    ```

    Then sync your environment:

    ```bash
    pnpm nizzy sync
    ```

    Restart the backend server for the new values to take effect.
  </Step>
</Steps>

### Connecting a Gmail account in Zero

Once Google OAuth is configured, connecting a Gmail account takes seconds.

<Steps>
  <Step title="Sign in to Zero">
    Open Zero at `http://localhost:3000` (or your production URL) and click **Sign in with Google**. This creates your Zero user account.
  </Step>

  <Step title="Connect additional accounts">
    Go to **Settings → Connections** and click **Add account**. You will be redirected through the Google OAuth flow again. Each additional account you connect becomes available in your unified inbox.

    <Note>
      You can connect multiple Gmail accounts to the same Zero user. Each account is stored as a separate entry in `mail0_connection` and appears as a distinct inbox in the sidebar.
    </Note>
  </Step>
</Steps>

### Google OAuth scopes

Zero requests the minimum scopes required to provide full email functionality:

| Scope | Purpose |
|-------|---------|
| `https://mail.google.com/` | Full Gmail access (read, compose, send, delete) |
| `https://www.googleapis.com/auth/gmail.modify` | Modify message labels and read/unread state |
| `https://www.googleapis.com/auth/userinfo.profile` | Display name and profile photo |
| `https://www.googleapis.com/auth/userinfo.email` | Email address for account identification |

## Microsoft Outlook (coming soon)

The Outlook driver (`OutlookMailManager`) is fully implemented and supports all core email operations — reading, composing, sending, labels, and attachments — using the Microsoft Graph API. The Microsoft OAuth provider is currently disabled in `auth-providers.ts` while final integration work is completed.

When Microsoft OAuth is enabled, it will require the following Graph API scopes:

| Scope | Purpose |
|-------|---------|
| `https://graph.microsoft.com/User.Read` | User profile information |
| `https://graph.microsoft.com/Mail.ReadWrite` | Read and modify mail |
| `https://graph.microsoft.com/Mail.Send` | Send mail |
| `offline_access` | Refresh tokens for persistent access |

The environment variables `MICROSOFT_CLIENT_ID` and `MICROSOFT_CLIENT_SECRET` are already defined in the environment schema and will be used once the provider is re-enabled.

<Note>
  Watch the [Zero GitHub repository](https://github.com/Mail-0/Zero) for updates on Microsoft Outlook support.
</Note>

## Multiple account connections

Zero is designed for a unified inbox experience. You can connect multiple email accounts from the same provider or different providers to a single Zero user account.

Each connection is stored independently in the `mail0_connection` table with its own OAuth tokens, email address, and provider ID. Tokens are refreshed automatically in the background — you do not need to reconnect accounts when access tokens expire.

## Troubleshooting

<AccordionGroup>
  <Accordion title="redirect_uri_mismatch error">
    The redirect URI registered in Google Cloud Console does not match the one the backend sends during the OAuth flow. Verify that:

    1. The URI in Google Cloud Console is exactly `http://localhost:8787/api/auth/callback/google` for development.
    2. `VITE_PUBLIC_BACKEND_URL` in your `.env` is set to `http://localhost:8787`.
    3. You ran `pnpm nizzy sync` after editing `.env`.
  </Accordion>

  <Accordion title="Access blocked: this app has not been verified">
    Your Google Cloud project is in Testing mode and the email you are signing in with is not on the test-user list. Go to [**APIs & Services → Audience**](https://console.cloud.google.com/auth/audience) and add your email under **Test users**.
  </Accordion>

  <Accordion title="Missing access or refresh tokens after sign-in">
    This usually means the OAuth consent screen was not configured with `accessType: offline`. Zero requests offline access by default — check that no custom `GOOGLE_REDIRECT_URI` is overriding the OAuth parameters.
  </Accordion>

  <Accordion title="Gmail API not enabled">
    If Zero signs in successfully but the inbox fails to load, confirm that both the Gmail API and the People API are enabled in your Google Cloud project under **APIs & Services → Enabled APIs**.
  </Accordion>
</AccordionGroup>
