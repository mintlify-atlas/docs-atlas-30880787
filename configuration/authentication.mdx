---
title: "Authentication in Zero: sign-in, sessions, and MCP tokens"
description: "Sign in to Zero with Google OAuth, find your session token in browser cookies, and use it to authenticate requests to the Zero MCP server."
---

Zero uses [Better Auth](https://better-auth.com) to manage authentication. Sign-in is handled entirely through Google OAuth — there is no username/password option. After a successful sign-in, Better Auth issues a session token stored in a browser cookie. That same token can be used to authenticate requests to the Zero MCP server.

---

## Signing in

<Steps>
  <Step title="Open Zero">
    Navigate to your Zero instance — `http://localhost:3000` for local development, or your production URL.
  </Step>

  <Step title="Click Sign in with Google">
    Zero redirects you to Google's OAuth consent screen. Sign in with the Google account you want to connect.

    <Note>
      If your Google Cloud project is in **testing** mode, only addresses you have explicitly added as test users can sign in. See [Email providers](/configuration/email-providers) for instructions on adding test users.
    </Note>
  </Step>

  <Step title="Grant permissions">
    Zero requests the following Google scopes:

    | Scope | Purpose |
    |-------|---------|
    | `https://mail.google.com/` | Full Gmail access for reading and sending mail |
    | `https://www.googleapis.com/auth/gmail.modify` | Modify labels, mark as read/unread |
    | `https://www.googleapis.com/auth/userinfo.profile` | Display name and profile photo |
    | `https://www.googleapis.com/auth/userinfo.email` | Email address for account identification |

    Click **Allow** to complete the flow.
  </Step>

  <Step title="You're in">
    Zero creates your account, stores the OAuth tokens securely, and redirects you to your inbox. Your session lasts 30 days and refreshes automatically every 3 days while you are active.
  </Step>
</Steps>

## Session tokens

Every sign-in creates a session stored in a browser cookie. The cookie name depends on the environment:

| Environment | Cookie name |
|-------------|-------------|
| `development` | `better-auth-dev.session_token` |
| `production` | `better-auth.session_token` |

### Finding your session token

<Steps>
  <Step title="Open browser developer tools">
    In Chrome or Firefox, open **DevTools** (`F12` or `Cmd+Option+I`) and navigate to the **Application** tab (Chrome) or **Storage** tab (Firefox).
  </Step>

  <Step title="Navigate to Cookies">
    In the left sidebar, expand **Cookies** and select the Zero domain (e.g., `http://localhost:3000`).
  </Step>

  <Step title="Copy the token value">
    Look for the cookie named `better-auth-dev.session_token` (development) or `better-auth.session_token` (production). Copy the **Value** column.

    ```
    better-auth-dev.session_token=<your_token_value>
    ```
  </Step>
</Steps>

<Warning>
  Your session token grants full access to your Zero account. Do not share it, commit it to version control, or store it in an insecure location. If you suspect your token is compromised, sign out to invalidate it.
</Warning>

## Using the session token with the MCP server

Zero exposes an MCP (Model Context Protocol) server that AI assistants can use to interact with your email. The MCP server authenticates requests using your session token as a Bearer token.

### MCP server URL

```
http://localhost:8787/mcp
```

For production deployments, replace `http://localhost:8787` with your backend URL.

### Authenticating requests

Pass your session token in the `Authorization` header:

```bash
Authorization: Bearer <your_session_token>
```

### Example: configuring Claude Desktop

Add Zero's MCP server to your Claude Desktop configuration (`claude_desktop_config.json`):

```json
{
  "mcpServers": {
    "zero-email": {
      "url": "http://localhost:8787/mcp",
      "headers": {
        "Authorization": "Bearer <your_session_token>"
      }
    }
  }
}
```

Replace `<your_session_token>` with the value you copied from the browser cookie.

<Tip>
  Session tokens expire after 30 days. If your MCP client stops working, retrieve a fresh token from the browser and update your configuration.
</Tip>

## Account linking and multiple providers

Better Auth is configured with account linking enabled. If you sign in with the same email address through different providers, Better Auth links the accounts automatically rather than creating duplicate users. Currently, `google` and `microsoft` are trusted providers for linking.

---

## Session lifetime

| Setting | Value |
|---------|-------|
| Session duration | 30 days |
| Refresh interval | Every 3 days of activity |
| Cookie scope | Cross-subdomain (controlled by `COOKIE_DOMAIN`) |

Sessions are stored in Redis (via the Upstash HTTP client) for fast lookups and in the `mail0_session` database table for persistence.
